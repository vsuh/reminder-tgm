# API

Приложение предоставляет REST API для управления расписаниями (а также набор обычных HTML-роутов веб-интерфейса). Все эндпоинты реализованы в `web/app.py`.

## Базовый URL

`http://host:<TLCR_FLASK_PORT>`

По умолчанию порт `7999` (при локальном запуске) или `7878` (в Docker-образе, см. `start_web_service.sh`).

## Аутентификация

Если в окружении заданы **обе** переменные `TLCR_WEB_USER` и `TLCR_WEB_PASSWORD`, доступ ко всем роутам, кроме `/login`, `/logout`, `/version` и `/test`, требует авторизации.

Аутентификация сессионная (cookie), а не HTTP Basic Auth и не токен в заголовке:

1. Выполните `POST /login` с полями формы `username` и `password`.
2. При успехе сервер выставит cookie сессии и сделает редирект.
3. Дальнейшие запросы (в том числе к JSON-эндпоинтам) должны передавать эту cookie.

Если хотя бы одна из переменных `TLCR_WEB_USER`, `TLCR_WEB_PASSWORD` не задана — авторизация отключена, и все роуты доступны без ограничений.

```bash
# Пример входа и сохранения cookie для последующих запросов curl
curl -c cookies.txt -X POST http://host:7878/login \
  -d "username=admin" -d "password=secret"

curl -b cookies.txt http://host:7878/schedules
```

## Формат данных

Все JSON-эндпоинты принимают и возвращают `Content-Type: application/json`. Ошибки валидации возвращаются с кодом `400` и телом вида:

```json
{
  "code": 400,
  "name": "Bad Request",
  "description": "текст ошибки"
}
```

## Расписания (schedules)

### Объект Schedule

| Поле | Тип | Описание |
|---|---|---|
| `id` | integer | Идентификатор расписания (только в ответах) |
| `cron` | string | `cron`-выражение (5 полей: минута час день месяц день_недели). Минутная часть при проверке срабатывания игнорируется |
| `message` | string | Текст уведомления. Может начинаться с `ДР` (день рождения) или `#!` (shebang-путь к JSON-файлу) — см. README |
| `modifier` | string | Модификатор периодичности: `""`, `d/n`, `w/n` или `YYYYMMDD>d/n` / `YYYYMMDD>w/n` |
| `chat_id` | integer | ID записи чата (внешний ключ на таблицу `chats`, **не** Telegram chat id напрямую) |
| `ntfy_id` | integer \| null | ID записи канала ntfy (внешний ключ на таблицу `ntfy_channels`), опционально |
| `last_fired` | string (timestamp) \| null | Когда расписание последний раз сработало (только в ответах) |

### `GET /schedules`

Возвращает список всех расписаний.

**Аутентификация:** требуется (если включена).

**Ответ `200`:**

```json
[
  {
    "id": 1,
    "cron": "1 19 * * *",
    "message": "note 1",
    "modifier": "20250526>d/4",
    "last_fired": "2025-05-30 19:01:00",
    "chat_id": 1,
    "ntfy_id": null
  }
]
```

---

### `POST /schedules`

Создаёт одно новое расписание.

**Тело запроса:**

```json
{
  "cron": "0 9 * * *",
  "message": "Полить цветы",
  "modifier": ""
}
```

| Поле | Обязательное | Примечание |
|---|---|---|
| `cron` | да | должно быть валидным cron-выражением |
| `message` | да | — |
| `modifier` | нет | по умолчанию `""` |

> **Внимание:** в отличие от `/schedules_all`, этот роут **не принимает** `chat_id` из тела запроса — функция `add_schedule` вызывается без него. Перед использованием убедитесь, что в таблице `chats` есть хотя бы одна запись, иначе вставка завершится ошибкой на уровне БД.

**Ответы:**

- `201 Created` — `{"status": "success"}`
- `400 Bad Request` — неверный формат данных, невалидный `cron` или `modifier`

---

### `POST /schedules_all`

Массовое создание расписаний из JSON-массива. Используется, в частности, для первоначального импорта (см. `static/dataschedules.json`).

**Тело запроса:** JSON-массив объектов в том же формате, что и для `POST /schedules`.

```json
[
  {
    "cron": "1 19 * * *",
    "message": "note 1",
    "modifier": "20250526>d/4"
  },
  {
    "cron": "0 13 * * 5",
    "message": "note 2",
    "modifier": ""
  }
]
```

**Пример:**

```bash
curl -X POST -H "Content-Type: application/json;charset=utf-8" \
  --data-binary @dataschedules.json \
  http://host:7878/schedules_all
```

**Ответы:**

- `201 Created` — `{"status": "success"}`
- `400 Bad Request` — если передан не массив, массив пуст, либо хотя бы один элемент не прошёл валидацию (обработка останавливается на первой ошибке)

---

### `POST /schedules/<id>`

Полностью обновляет существующее расписание (JSON-эндпоинт, отличается от HTML-формы `/edit/<id>`).

**Тело запроса:**

```json
{
  "cron": "0 9 * * *",
  "message": "Полить цветы",
  "modifier": "",
  "chat_id": 1
}
```

| Поле | Обязательное |
|---|---|
| `cron` | да |
| `message` | да |
| `chat_id` | да |
| `modifier` | нет, по умолчанию `""` |

**Ответы:**

- `200 OK` — `{"status": "success"}`
- `400 Bad Request` — неверный формат данных или невалидный `cron`/`modifier`

---

### `DELETE /schedules/<id>`

Удаляет расписание по идентификатору.

**Ответ `200`:**

```json
{"status": "deleted"}
```

> Эндпоинт идемпотентен и не возвращает `404`, даже если расписание с таким `id` не существовало.

---

### `GET /export`

Выгружает все расписания в виде JSON-файла (вложение `schedules_export.json`), пригодного для последующей загрузки через `POST /schedules_all`.

```bash
curl -o schedules_export.json http://host:7878/export
```

## HTML-роуты веб-интерфейса

Эти роуты возвращают HTML-страницы и предназначены для использования через браузер, но могут быть полезны и при автоматизации (например, формы можно эмулировать через `curl -d`).

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/test` | Проверка доступности — возвращает `ok`. Без авторизации |
| `GET` | `/version` | Версия приложения: `{"version": "<TAG>"}`. Без авторизации |
| `GET`, `POST` | `/login` | Страница входа |
| `GET` | `/logout` | Выход из сессии |
| `GET`, `POST` | `/` | Список расписаний (`GET`); создание расписания через форму (`POST`, поля `cron`, `message`, `modifier`, `chat_id`, `ntfy_id`) |
| `GET`, `POST` | `/edit/<id>` | Форма редактирования расписания |
| `POST` | `/schedules/<id>/delete` | Удаление расписания через HTML-форму (редирект на `/`) |
| `GET` | `/list/<id>` | Предпросмотр ближайших `TLCR_LIST_ITEMS` срабатываний расписания, с подстановкой возраста для записей о днях рождения |
| `GET` | `/message/<id>` | Просмотр содержимого JSON-файла, подключённого через shebang-сообщение (`#!...`) |
| `GET`, `POST` | `/chats` | Список чатов (`GET`); добавление чата (`POST`, поля `name`, `chat_id`) |
| `GET` | `/chats/delete/<id>` | Удаление чата и всех связанных с ним расписаний |
| `GET`, `POST` | `/ntfy` | Список каналов ntfy (`GET`); добавление канала (`POST`, поля `name`, `url`, `title`) |
| `GET`, `POST` | `/ntfy/edit/<id>` | Редактирование канала ntfy |
| `GET` | `/ntfy/delete/<id>` | Удаление канала ntfy (у связанных расписаний `ntfy_id` сбрасывается в `NULL`) |
| `GET` | `/drop_db` | Полная переинициализация базы данных (удаление и пересоздание таблиц). **Деструктивная операция** |

### Поля формы чата (`POST /chats`)

| Поле | Описание |
|---|---|
| `name` | Произвольное название чата для отображения в UI |
| `chat_id` | Реальный Telegram chat id (число, может быть отрицательным для групп) |

### Поля формы канала ntfy (`POST /ntfy`, `POST /ntfy/edit/<id>`)

| Поле | Обязательное | Описание |
|---|---|---|
| `name` | да | Название канала |
| `url` | да | Полный URL топика ntfy |
| `title` | нет | Заголовок уведомления (только ASCII, иначе будет проигнорирован при отправке) |

## Валидация расписаний

Все эндпоинты, создающие или обновляющие расписание (`/schedules`, `/schedules_all`, `/schedules/<id>` (POST), HTML-формы `/` и `/edit/<id>`), используют одну и ту же проверку:

1. Наличие полей `cron` и `message`.
2. Валидность `cron`-выражения — проверяется библиотекой `croniter`.
3. Если указан `modifier` — он проверяется через `VCron.check_modifier`. Допустимые форматы: `""`, `d/n`, `w/n`, `YYYYMMDD>d/n`, `YYYYMMDD>w/n` (подробнее — в README, раздел «Модификаторы расписания»).

При несоблюдении любого из условий возвращается `400 Bad Request` с описанием ошибки в поле `description`.

---

## Коды ошибок

| Код | Когда возникает |
|---|---|
| `400` | Невалидные данные запроса (отсутствуют поля, неверный `cron`/`modifier`, пустой массив в `/schedules_all` и т.п.) |
| `404` | Расписание / ntfy-канал с указанным `id` не найден (`/edit/<id>`, `/list/<id>`, `/ntfy/edit/<id>`) |
| `500` | Внутренняя ошибка (например, при удалении чата/канала или переинициализации БД) |

## Примеры

### Получить список расписаний

```bash
curl http://host:7878/schedules
```

### Создать расписание (без указания чата)

```bash
curl -X POST -H "Content-Type: application/json" \
  -d '{"cron": "0 9 * * *", "message": "Полить цветы", "modifier": ""}' \
  http://host:7878/schedules
```

### Обновить расписание с id=1

```bash
curl -X POST -H "Content-Type: application/json" \
  -d '{"cron": "0 9 * * 1", "message": "Полить цветы (по понедельникам)", "modifier": "", "chat_id": 1}' \
  http://host:7878/schedules/1
```

### Удалить расписание с id=1

```bash
curl -X DELETE http://host:7878/schedules/1
```

### Массовая загрузка расписаний из файла

```bash
curl -X POST -H "Content-Type: application/json;charset=utf-8" \
  --data-binary @dataschedules.json \
  http://host:7878/schedules_all
```

### Выгрузить текущие расписания в файл

```bash
curl -o schedules_export.json http://host:7878/export
```

### Проверить версию приложения

```bash
curl http://host:7878/version
# {"version": "v1.4.8"}
```
